0.1 术语说明

术语说明举例
业务链路围绕一个客户、商机或项目，从前端销售到后端交付、收入、售后的完整业务路径。线索 → 商机 → 合同 → 项目 → 交付 → 验收 → 收入
链路实例某个具体客户、商机或项目实际形成的一条链路记录。“XX 客户 5G 消息项目”的完整推进轨迹。
事件日志系统内每一次关键业务动作形成的标准化记录，是链路还原和效能分析的数据基础。创建线索、转商机、提交合同、项目上线、初验通过。
标准链路系统预设或管理员配置的规范业务流程。商机必须经过“合作跟进 → 合同签署中 → 签约成功”。
实际链路根据事件日志还原出的真实业务流转过程。某商机实际经历了 3 次退回、2 次补充资料后才签约。
流程偏差实际链路与标准链路、节点顺序、资料要求、时限要求不一致的情况。项目未上线却提交初验；已终验但缺少终验报告。
返工同一节点或相邻节点重复退回、重复修改、重复补充材料的情况。验收资料连续退回 3 次。
1

需求概述

用一页说明需求是什么、解决什么问题、交付什么能力。

需求名称

ToB 全链路效能中心

需求来源

ToB 系统现代化建设 / 全链路流程治理 / 经营效能分析

优先级

P0 / P1 核心经营分析与流程治理能力

一句话说明：通过新增“全链路效能中心”，自动沉淀 ToB 业务关键动作日志，还原客户、线索、商机、项目、交付、收入等真实业务链路，识别耗时瓶颈、流程偏差、临期超时、资料缺失和返工问题，帮助业务管理者从结果管理转向过程治理。
全链路事件日志效能分析预警闭环偏差检测

1.1 建设原则

不增加一线填报负担

优先基于现有业务操作自动生成事件日志，不要求用户额外维护“流程数据”。

先看清，再治理

一期重点还原链路、计算耗时、识别异常，不直接做复杂仿真和算法平台。

业务可解释

看板、预警、诊断必须能下钻到具体客户、商机、项目、人员、节点和操作记录。

2

需求背景与目标

先讲问题，再讲目标，避免直接堆功能。

2.1 现状 / 问题

1. 线索、商机、项目、交付、收入、售后分别在不同模块维护，单个对象可查看，但缺少跨对象的完整链路视角。

2. 当前更多依赖状态字段反映进度，无法清晰判断每个阶段实际停留多久、退回几次、是否跳过关键节点。

3. 管理者想看“哪里慢、哪里堵、哪里风险高”，但现有数据多为结果型统计，过程型指标不足。

4. 临期、超时、资料缺失、状态异常等问题发现滞后，通常依赖人工排查、会议追踪和线下催办。

5. 标准流程、节点时限、必备资料、责任角色缺少统一配置入口，导致不同项目、不同部门执行口径不一致。

2.2 建设目标

1. 建立统一的业务事件日志，支撑全链路还原、耗时计算、偏差检测和审计追溯。

2. 建设全链路效能总览，按阶段、产品、区域、行业、部门、人员分析流程效率。

3. 建设链路实例追踪，支持从客户、商机、项目任一对象进入，查看完整实际流转轨迹。

4. 建设流程偏差和预警能力，自动识别跳节点、顺序异常、长期无进展、资料缺失、返工等问题。

5. 建设链路规则配置中心，沉淀标准链路、节点 SLA、必备材料、责任角色和预警规则。

2.3 建设目标与指标

效率目标减少人工统计和会议追踪成本手工整理全链路报表耗时下降 50% 以上按月统计产品/运营人员报表准备时间

3

需求范围与边界

明确本期做什么、不做什么、依赖什么。

本期范围

1. 业务事件日志标准设计与关键模块埋点。

2. 链路模板、节点、SLA、必备资料、预警规则配置。

3. 全链路效能总览、链路分析、链路实例列表与详情。

4. 流程偏差检测与预警中心。

5. 消息通知、处理闭环、操作日志、导出。

不在本期范围

1. 复杂 BPMN 拖拽建模和流程仿真预测。

2. 通用流程挖掘算法平台和自定义流程查询语言。

3. RPA 自动执行、跨系统自动改数。

4. 历史多年度数据全量准确还原。

5. 移动端完整适配，仅保证核心详情可查看。

前置依赖

1. 客户、拜访、线索、商机、合同、项目、交付、收入模块关键字段明确。

2. 组织、人员、角色、数据权限可用。

3. 消息中心支持站内信/待办/邮件等通道。

4. 各模块关键动作可接入事件日志。

5. 历史数据映射规则由业务确认。

4.1 典型使用场景

管理例会前

管理者查看本月各阶段耗时、超时事项、重点风险客户和项目，直接形成会议议题。

项目推进中

交付经理从项目详情进入链路追踪，确认上线、初验、终验、收入节点是否按计划推进。

流程治理复盘

运营人员筛选多次退回、长期无进展、资料缺失项目，形成流程优化建议和责任闭环。

5.3 事件生成流程

步骤触发动作系统处理输出结果异常处理
1用户在原业务模块完成关键操作原模块调用事件日志服务，传入对象、动作、状态、时间、操作人等信息。生成业务事件日志事件生成失败不阻断主业务，但需记录补偿任务。
2事件入库系统根据对象类型、对象 ID、客户 ID、关联商机/项目 ID 判断归属链路实例。绑定链路实例无法匹配时进入“待匹配事件池”。
3链路计算任务执行按事件时间排序，计算节点开始、完成、停留时长、是否超时。更新链路实例、节点记录重复事件按幂等规则处理。
4规则引擎执行匹配标准链路、SLA、资料要求、偏差规则、预警规则。生成偏差记录或预警记录规则配置异常时记录系统告警。
5预警推送根据规则发送站内信、待办、邮件或短信。责任人收到待处理事项发送失败支持重试和日志追踪。

9.2 节点状态

节点状态说明触发条件展示规则
未到达标准链路中存在，但实际还未发生对应事件。前序节点未完成。灰色展示。
进行中已到达但未完成。发生节点开始事件，未发生完成事件。蓝色展示，计算当前停留时长。
已完成节点正常完成。发生节点完成事件。绿色展示，展示耗时。
超时节点停留时长超过 SLA。当前时间/完成时间 - 开始时间 > SLA。橙色或红色展示。
跳过必经节点未发生，但后续节点已发生。后序节点完成且本节点无事件。红色展示，生成偏差。
异常节点存在顺序、状态、资料、责任异常。命中偏差规则。红色展示，显示异常原因。

9.4 业务规则补充

状态不可手动修改：链路实例、节点、预警状态原则上不允许前端直接编辑，只能由业务动作、事件日志、规则计算和处理动作驱动。
事件幂等：同一来源模块、同一业务对象、同一动作、同一操作时间、同一业务唯一键只能生成一条有效事件。
历史回算：上线前可选择近 6-12 个月历史数据进行一次性回算；历史事件缺失导致的偏差不纳入正式考核，仅用于参考分析。

14.4 回退方案

回退原则：全链路效能中心不影响原业务模块主流程，若上线后出现异常，可先关闭看板入口、预警扫描任务和消息推送，不回滚原业务数据。
异常类型回退措施数据处理恢复条件
事件写入异常关闭事件接入或降级为异步补偿失败事件进入补偿队列接口稳定后重放补偿
预警误报严重暂停预警扫描和消息发送已生成预警标记为待复核规则调整并灰度验证后恢复
看板性能异常关闭大范围查询和导出保留事件日志和实例数据优化聚合与缓存后恢复
权限异常临时关闭菜单入口不删除业务数据权限矩阵验证通过后恢复
